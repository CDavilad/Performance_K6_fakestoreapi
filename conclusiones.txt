====================================================================
Herramienta: k6 v0.54+

1. RESUMEN EJECUTIVO
--------------------
Se ejecutaron pruebas de humo, carga y estrés sobre el endpoint de autenticación
de fakestoreapi.com. El servicio superó todos los criterios de aceptación en los
tres tipos de prueba (0 % errores, p95 < 1 500 ms, ≥ 20 TPS). Se confirmó que la
API pública aplica rate-limiting en torno a 50 TPS, lo que impidió encontrar
un punto de quiebre real bajo estrés.

6. OBSERVACIONES TÉCNICAS
--------------------------
- El executor `ramping-arrival-rate` es el más adecuado para este escenario
  porque garantiza la tasa de peticiones independientemente de la latencia,
  permitiendo validar con precisión el requisito de 20 TPS.
- La API devuelve HTTP 201 con token para credenciales válidas y HTTP 401
  para credenciales incorrectas. El status 201 es correcto según la semántica
  REST de fakestoreapi; los checks de la suite aceptan cualquier código 2xx.
- La credencial "user/passwd" del enunciado es un ejemplo ilustrativo y no
  corresponde a un usuario real de la API; se excluyó del CSV después de
  confirmar el 401 en la primera ejecución.
- `SharedArray` de k6 permite cargar el CSV una sola vez en memoria y
  compartirlo entre todos los VUs sin duplicación, lo cual es la práctica
  recomendada para conjuntos de datos de entrada.
- Para entornos controlados (staging/producción propia), se recomienda
  combinar k6 con un output de InfluxDB + Grafana para visualización
  en tiempo real de métricas TPS, percentiles y error rates.


7. RECOMENDACIONES
------------------
- Si el servicio es propio, implementar connection pooling y caché de
  tokens para reducir la carga del endpoint de login bajo alta concurrencia.
- Agregar pruebas de soak (30+ minutos a 20 TPS) para detectar memory leaks
  o degradación por acumulación de sesiones.
- Automatizar la ejecución en CI/CD para detectar regresiones de rendimiento
  ante cambios en el servicio de autenticación.
